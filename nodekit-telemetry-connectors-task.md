# Реализация новых коннекторов телеметрии в nodekit

## Контекст

В библиотеке `gravity-ui/nodekit` сейчас существует единственный адаптер телеметрии `clickhouse.ts`, который пишет данные напрямую в ClickHouse через HTTPS POST с SQL `INSERT` в теле запроса. Адаптер реализует батчинг через `prepareBatchedQueue`, поддерживает несколько таблиц, backlog, mirror в логи и инжектит host/timestamp в дефолтную таблицу `apiRequests`

Прямая запись в ClickHouse даёт минимум абстракции и накладывает на сервис ряд эксплуатационных ограничений:
- сервис должен знать сетевые координаты ClickHouse и владеть кредами на запись
- при пиках нагрузки и батчинге внутри Node.js процесса возможна потеря данных при рестартах подов
- единая точка отказа в виде ClickHouse напрямую завязана на доступность приложения для отправки телеметрии
- нет универсальной точки маршрутизации для добавления новых получателей телеметрии (Greenplum, S3, ML pipeline)

## Цель

Добавить в nodekit три альтернативных адаптера телеметрии в дополнение к существующему `clickhouse.ts` и провести исследование возможностей Unified Agent как универсального транспорта между приложением и хранилищами

Каждый адаптер должен повторять публичный контракт `prepareClickhouseClient` (фабрика принимает `ctx`, возвращает объект с методом `sendStats(data)` и `sendStats(tableName, data)`) для возможности безболезненной замены в существующих сервисах через конфигурацию

## Объём работ

### 1. Общий HTTP коннектор

Адаптер `http.ts` отправляет события телеметрии POST запросом на произвольный HTTP endpoint. Получателем может быть любой collector (sidecar-агент, edge-прокси, самописный приёмник, OpenTelemetry collector и т.д.), который дальше доставляет данные в конечное хранилище

Требования к реализации:
- POST на произвольный `http(s)://<host>:<port>/<path>`
- тело запроса в JSON, массив событий на сообщение (батч)
- произвольные HTTP заголовки для маршрутизации на стороне получателя (project, service и т.д.)
- имя таблицы/стрима пробрасывается в отдельном заголовке (`X-Telemetry-Table` по умолчанию, настраивается)
- батчинг и backlog аналогично существующему ClickHouse адаптеру
- ретраи с экспоненциальным backoff, поддержка отбрасывания при переполнении буфера

Конфигурация через переменные:
- `appTelemetryHttpUrl`
- `appTelemetryHttpHeaders` (пробрасываемые HTTP-заголовки)
- `appTelemetryHttpTableHeader` (имя заголовка с именем таблицы/стрима)
- `appTelemetryHttpBatchSize`, `appTelemetryHttpSendInterval`, `appTelemetryHttpBacklogSize`
- `appTelemetryHttpRequestTimeout`

Ответственность за дальнейшую доставку (в ClickHouse, Kafka/Logbroker, YDS, S3) переносится на конфигурацию collector-а. Инфраструктурные сценарии с Unified Agent в качестве получателя - частный случай, рассматриваются в исследовательской части

### 2. Kinesis коннектор для Yandex DataStreams

Адаптер `yds-kinesis.ts` отправляет события напрямую в Yandex DataStreams по Kinesis API через метод `PutRecords`

Документация на API: https://yandex.cloud/ru/docs/data-streams/kinesisapi/methods/putrecords

Требования к реализации:
- формирование запроса по схеме `PutRecords` с массивом `Records` (1-500 на запрос)
- кодирование `Data` в base64
- расчёт `PartitionKey` на основе `requestId` или `traceId` (для равномерного распределения по шардам)
- учёт лимитов: до 1 MB на запись, до 5 MB на запрос (включая `PartitionKey`)
- AWS Signature V4 либо IAM token аутентификация в зависимости от схемы доступа
- обработка `FailedRecordCount` в ответе с повторной отправкой только неуспешных записей
- адаптивный батчинг по количеству записей и суммарному размеру

Конфигурация:
- `appTelemetryYdsEndpoint`
- `appTelemetryYdsStreamName`
- `appTelemetryYdsAuth` (IAM token, static credentials или service account key)
- `appTelemetryYdsRegion`
- стандартные параметры батчинга

### 3. Логи с маркером телеметрии

Адаптер `logs-marker.ts` пишет события телеметрии в обычный логгер приложения (`ctx.log`) с явным маркером в JSON ключах для последующей фильтрации на стороне сборщика логов

Структура записи в логе:
```
{
  "_telemetry": true,
  "_telemetry_type": "api_request",
  "_telemetry_table": "apiRequests",
  "service": "...",
  "action": "...",
  ...поля события
}
```

Требования к реализации:
- запись через существующий механизм логирования без сетевых вызовов
- зарезервированные ключи `_telemetry`, `_telemetry_type`, `_telemetry_table` для надёжной фильтрации
- опциональный sampling для высокочастотных событий через `appTelemetryLogsSampleRate`
- уровень логирования настраивается (по умолчанию `info`)

Адаптер не делает батчинг и сетевую отправку - доставка данных в хранилище полностью на стороне инфраструктуры сбора логов через docker stdout, journald, file и далее через Unified Agent

## Исследовательская часть

Помимо реализации адаптеров необходимо проверить и задокументировать в отдельном Wiki-разделе несколько технических вопросов по Unified Agent

### 3.1. Может ли Unified Agent сам отправлять данные в Kinesis через HTTP output

Документация на output: https://docs.yandex-team.ru/monium/agent/configuration/outputs#http

По первичному анализу:
- HTTP output поддерживает произвольный URL, кастомные headers через `!expr` синтаксис, IAM аутентификацию через `cloud_meta`/`token_agent`/`jwt`
- но тело запроса формируется как сырой payload сообщения, без преобразования в формат `PutRecords` (массив Records с base64 Data и PartitionKey)
- значит готовый HTTP output без дополнительной обработки в Kinesis API писать не сможет

Что нужно проверить:
- получится ли через `lua_message_filter` сформировать корректное тело `PutRecords` (base64 кодирование, оборачивание в JSON массив, расчёт `PartitionKey`)
- альтернатива через `subprocess` фильтр и внешний скрипт-преобразователь
- работоспособность batch фильтра с lenval/protoseq форматом совместно с lua-преобразованием
- корректная работа подписи запроса (AWS SigV4 либо IAM Bearer token) на стороне UA

Если ответ положительный, это снимает потребность в адаптере 2 для подов внутри инфраструктуры с UA

### 3.2. Может ли Unified Agent фильтровать строки телеметрии из JSON логов

Документация на фильтры: https://docs.yandex-team.ru/monium/agent/configuration/filters

Задача: на хосте поднят `file_input` или `journald` input для общих логов приложения, среди записей встречаются строки с маркером `_telemetry: true` (см адаптер 3), эти строки нужно вырезать из общего потока логов и отправить в отдельный output (ClickHouse, YDS, Logbroker)

Что нужно проверить:
- комбинация `parse_json` фильтра для разбора JSON и `match` фильтра для отбора по метаданным сообщения
- возможность использовать `enable_if` для условной маршрутизации в разные channels
- необходимо ли строить два независимых channel с одним input (общие логи минус телеметрия) и (только телеметрия) либо это решается через split в pipeline

Ожидаемый результат - готовая конфигурация UA, демонстрирующая разделение потока

### 3.3. Может ли Unified Agent писать в YDB Topics (Yandex DataStreams)

YDS под капотом построен на YDB Topics, доступ возможен и по Kinesis API и по нативному PQ протоколу

Что нужно проверить:
- работает ли output `logbroker` с параметрами `use_ydb_discovery: true`, `use_ssl: {}`, IAM auth и cloud-эндпоинтом `yds.serverless.yandexcloud.net:2135`
- ограничения по сравнению с Kinesis API (квоты, протокол, codec)
- сравнение нативного PQ с Kinesis API по производительности и удобству эксплуатации

Ожидаемый результат - рабочая конфигурация UA с записью в YDS через `logbroker` output на тестовом стенде

## Тестирование

Цель тестирования - убедиться в надёжной доставке телеметрии и в том, что замена одного адаптера на другой не ломает существующие сервисы. Тесты строятся в четыре слоя: контрактные, юнит-тесты на каждый адаптер, интеграционные и нагрузочные/устойчивость к отказам

Инструментарий: уже подключённый в проекте Jest (`jest.config.js`), `nock` для мока HTTP, `jest.useFakeTimers` для контроля над тикером батчинга (аналогично `src/tests/utils/batch.test.ts`), docker-compose для интеграционных тестов

Все тесты адаптеров складываются в `src/tests/telemetry/<adapter>.test.ts` либо в `src/lib/telemetry/__tests__/`, единообразно с существующей структурой

### 4.1. Контрактные тесты (общие для всех адаптеров)

Файл `src/tests/telemetry/contract.test.ts` параметризованный по адаптерам. Каждый адаптер прогоняется через единый набор проверок, что гарантирует взаимозаменяемость

Проверки:
- фабрика возвращает функцию (или объект) с сигнатурой `sendStats(data)` и `sendStats(tableName, data)`
- при отсутствии обязательной конфигурации фабрика возвращает no-op (как `prepareClickhouseClient` сейчас) и не бросает исключений
- вызов `sendStats` с неизвестным `tableName` логирует ошибку через `ctx.logError`, но не падает
- автоматическое обогащение `host` и `timestamp` для дефолтной таблицы `apiRequests` (сохраняется поведение текущего ClickHouse адаптера)
- секреты (`appTelemetryChAuth`, IAM токен, AWS keys, headers с авторизацией) не появляются в `ctx.log`/`ctx.logError` ни при штатной работе, ни при ошибках сериализации/сети (snapshot вызовов логгера матчится против regex с маской)
- метрики `sent`, `failed`, `dropped`, `backlogSize` доступны через возвращаемый объект и инкрементируются согласно сценарию
- graceful shutdown через `client.shutdown()` / `client.flush()` дренирует очередь: все накопленные события улетают в `fn` до резолва промиса

### 4.2. Юнит-тесты на адаптер 1 (`http.ts`)

Файл `src/tests/telemetry/http.test.ts`

Характер: чистые юнит-тесты на `axios` через `nock` или `jest.mock('axios')`, fake timers

Кейсы:
- batching: 250 событий с `batchSize=100` приводят ровно к 3 POST запросам после `advanceTimersByTime(tickInterval)`
- формат тела: JSON массив событий, заголовки `Content-Type: application/json`, кастомные `appTelemetryHttpHeaders` пробрасываются в запрос
- маршрутизация по таблицам: при вызове `sendStats('customTable', data)` в headers попадает `X-Telemetry-Table: customTable` (имя заголовка настраивается через `appTelemetryHttpTableHeader`)
- ретраи с экспоненциальным backoff: первый ответ 503 → ретрай через X мс, второй 503 → 2X мс, успех на третьем; счётчик попыток лимитирован (по аналогии с `does not retry more than three times` из `batch.test.ts`)
- 4xx ошибки не ретраятся, событие уходит в `dropped` с логированием
- backlog overflow: при `backlogSize=50` и 200 событиях с зависающим UA отбрасывается ровно 150, метрика `dropped=150`
- timeout запроса (UA не отвечает) обрабатывается как retryable failure
- shutdown во время in-flight запроса ждёт его завершения

### 4.3. Юнит-тесты на адаптер 2 (`yds-kinesis.ts`)

Файл `src/tests/telemetry/yds-kinesis.test.ts`

Кейсы:
- сериализация `PutRecords`: `Data` корректно base64-кодируется, `PartitionKey` берётся из `requestId`/`traceId`, при отсутствии - детерминированный хэш от полезной нагрузки (а не случайный, для воспроизводимости в тестах)
- лимиты: батч из 600 записей разбивается на 2 запроса (≤500 в каждом); запись >1 MB отбрасывается с логированием в `dropped` и не блокирует остальные; суммарный размер запроса не превышает 5 MB (тест с искусственно крупными payload)
- частичный успех: ответ `{FailedRecordCount: 2, Records: [...]}` с двумя ошибочными элементами приводит к повторной отправке только этих двух; успешные не дублируются
- ProvisionedThroughputExceededException на уровне записи → retry с backoff
- аутентификация: тест на формирование AWS SigV4 подписи (детерминированный input → ожидаемый `Authorization` header), отдельный тест на IAM Bearer token mode
- ротация IAM токена при 401/403 (если используется внешний token provider - mock метода `getToken`)
- секреты не утекают: при сериализации запроса в лог попадает только маска `[REDACTED]` вместо ключей/токенов

### 4.4. Юнит-тесты на адаптер 3 (`logs-marker.ts`)

Файл `src/tests/telemetry/logs-marker.test.ts`

Характер: тесты на мок `ctx.log`, без сети и таймеров

Кейсы:
- структура записи содержит `_telemetry: true`, `_telemetry_type`, `_telemetry_table` в корне JSON
- nested namespace (либо префикс) применяется к пользовательским полям, чтобы не конфликтовать с возможным `_telemetry` из других источников (тест на коллизию: входная запись с собственным полем `_telemetry` не должна затереть маркер)
- sampling: при `appTelemetryLogsSampleRate=0.1` и 10000 вызовах фактическое количество записей в логе попадает в доверительный интервал около 1000 (детерминированный seed для PRNG в тесте)
- уровень логирования настраивается (`info` по умолчанию, override через конфиг)
- адаптер синхронный, не использует сеть и не создаёт таймеров (проверка через `jest.getTimerCount() === 0`)
- ключи `_telemetry*` зарезервированы: при попытке пользователя передать `_telemetry: false` в данных это значение игнорируется, маркер остаётся `true`

### 4.5. Регрессионные тесты на существующий `clickhouse.ts`

Текущего тестового покрытия адаптера в репозитории нет (поиск по `src/tests/` не показывает `clickhouse.test.ts`). До любой переделки на общий контракт необходимо зафиксировать существующее поведение:
- escape спецсимволов в строках (`'`, `\`) в `prepareInsertValues`
- генерация SQL `INSERT` для дефолтной таблицы `apiRequests` с автоматическим `host`/`timestamp`
- mirror в логи при `appTelemetryChMirrorToLogs=true`
- редактирование пароля в логах ошибок (`error.config.params.password = '[REDACTED]'`)
- обработка `incorrect` для неизвестного типа колонки

Это страховка от ломки публичного контракта при рефакторинге под общую фабрику

### 4.6. Интеграционные тесты

Каталог `src/tests/telemetry/integration/`, запускаются отдельным npm скриптом `npm run test:integration` (не входят в основной `npm test`, требуют docker)

Стенды:
- **HTTP collector**: docker-compose с простым HTTP-приёмником (echo-сервер либо Unified Agent с HTTP input + file output под исследовательскую часть), бинд на 127.0.0.1. Тест шлёт N событий через адаптер, ждёт flush, читает принятые данные и проверяет 1:1 соответствие
- **YDS Kinesis**: либо `localstack/localstack` с kinesis сервисом, либо реальный тестовый стрим (под `APP_TEST_YDS_ENDPOINT` env). Тест шлёт записи, делает `GetRecords` и проверяет полный набор + правильность распределения по шардам через `PartitionKey`
- **Logs marker → UA filter**: запускаем приложение, пишущее через `logs-marker` в stdout, UA с конфигом из исследовательской части 3.2 разделяет потоки. Проверяем, что строки с маркером не попадают в общий output и попадают в телеметрический

Интеграционные тесты помечаются `describe.skip` если соответствующая env переменная не задана - в CI они прогоняются по расписанию/тегу, в локальной разработке опционально

### 4.7. Тесты на сценарии отказов и устойчивость

Отдельный файл `src/tests/telemetry/resilience.test.ts`, прогоняется для каждого сетевого адаптера (UA HTTP, YDS Kinesis)

Сценарии:
- получатель полностью недоступен (`ECONNREFUSED`) в течение N тиков - данные удерживаются в backlog до лимита, дальше дропаются с метрикой, при восстановлении доставляются оставшиеся
- получатель отвечает медленно (искусственная задержка) - backlog заполняется, новые события дропаются, не блокируют горячий путь `sendStats` (тест на максимальное время выполнения `sendStats` - микросекунды)
- частичные сетевые ошибки (50% ответов 5xx) - eventually consistency: за достаточное время все события доставлены
- рестарт процесса посреди работы: вызов `process.emit('SIGTERM')` приводит к `flush` очередей за tunable timeout, после чего процесс завершается; недоставленные события логируются
- проверка что `sendStats` никогда не бросает исключение наружу (fuzz-like тест с произвольными входами: undefined, null, циклические объекты, BigInt, символы)

### 4.8. Нагрузочные/performance тесты

Отдельно от Jest, в `src/tests/telemetry/benchmark/` через `tinybench` или простой `console.time`-харнесс. Не блокируют CI, запускаются вручную перед релизом

Метрики:
- throughput `sendStats` (events/sec) при разных размерах payload (100B, 1KB, 10KB) - все три адаптера должны держать ≥50k events/sec на горячем пути (push в очередь), фактическая сетевая отправка может быть медленнее
- сравнение memory footprint при `backlogSize=10000` событий
- задержка от вызова `sendStats` до подтверждённой доставки (p50/p95/p99) для каждого адаптера

Результаты складываются в Wiki вместе с матрицей выбора адаптера, чтобы команды могли осознанно выбирать вариант

### 4.9. Покрытие и CI

- целевой code coverage по `src/lib/telemetry/` - не ниже 85% lines/branches, проверяется через `jest --coverage`
- юнит-тесты + контрактные включаются в основной `npm test` и блокируют PR
- интеграционные и resilience-тесты идут отдельной job в CI с docker-сервисами, могут быть отмечены как required для релизных веток
- нагрузочные - только ручной запуск, отчёт прикладывается к релизу при существенных изменениях в адаптерах

## Точки внимания

- сохранить обратную совместимость с существующим `prepareClickhouseClient`
- единая модель ошибок и метрик во всех адаптерах (счётчики `sent`, `failed`, `dropped`, `backlogSize`)
- graceful shutdown с дренажом очередей перед остановом процесса
- секреты (IAM токен, AWS keys, credentials) не должны попадать в логи даже при ошибках сериализации запроса
- для адаптера 3 необходимо предусмотреть, что обычные логи приложения уже могут содержать ключи `_telemetry` от других источников - использовать nested namespace или префикс для надёжного разграничения
